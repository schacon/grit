## Failure observed
Running `bash tests/t1012-read-tree-df.sh` failed `3-way (1..3)`:
- expected unmerged stage entries (`1/2/3`) for D/F conflict cases,
- observed stage-0 entries in place of expected conflict staging,
- one case aborted with `Not a directory (os error 20)` due to incorrect
  conflict preservation behavior in update step.

## Root cause
`read-tree -m` three-way merge logic in `grit/src/commands/read_tree.rs`
was preserving all current stage-0 entries that were outside exact merge-path
keys. This leaked existing stage-0 entries for paths that were parents/children
of merge paths (directory/file conflicts), preventing proper unmerged stage
construction and resulting in stage collapse / checkout errors.

Specifically, preservation filtered only by exact path membership:
- preserve stage-0 when `!merge_paths.contains(&entry.path)`.

For D/F conflicts this is insufficient; parent/child collisions must also be
considered “in merge scope”.

## Fix implemented
File changed: `grit/src/commands/read_tree.rs`

1. Added helper:
   - `path_in_df_conflict_scope(path, merge_paths) -> bool`
   - Returns true if `path` exactly matches, is parent of, or is child of any
     merge path (`paths_conflict_for_df` aware).

2. Updated three-way merge preservation logic:
   - preserve prior stage-0 entries only when they are outside D/F conflict
     scope of merge paths.
   - this prevents stale stage-0 entries from suppressing required stage
     `1/2/3` conflict entries.
