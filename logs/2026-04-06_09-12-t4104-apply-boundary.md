### Root cause analysis
- `git apply` hunk placement logic treated add-only (`old_count=0`) hunks as “insert before `old_start - 1`” instead of Git-compatible insertion semantics.
  - This misplaced insertions for:
    - `@@ -1,0 +2 @@` (insert-after-line-1 case),
    - `@@ -24,0 +25 @@` (append-at-end case).
- Add-only hunks with no old-side context could still apply when patch preimage blob IDs did not match the current content, causing false positives in boundary checks like the “two lines” case.

### Implementation
- Updated hunk placement logic in `grit/src/commands/apply.rs`:
  - introduced `nominal_hunk_start(...)` with Git-compatible handling for `old_count=0` insertion hunks;
  - kept normal behavior for non-zero old-count hunks.
- Added `verify_old_oid_matches_content(...)` and wired it into:
  - worktree apply path,
  - index apply path,
  - `--check` preflight path,
  so add-only hunks require preimage OID compatibility where expected.
- Ensured reject mode behavior remains unchanged:
  - preimage OID verification is skipped for `--reject` so partial-apply/reject flow still works.
