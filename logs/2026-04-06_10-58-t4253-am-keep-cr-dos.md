### Initial observations
- `grit am` currently rejects both `--keep-cr` and `--no-keep-cr` as unknown arguments.
- Even without explicit flags, applied patch content drops CR (`\r`) bytes versus expected keep-cr behavior paths.
- Later tests fail cascadingly because option parsing breaks resume/continue scenarios, leaving branch/worktree state inconsistent for subsequent assertions.

### Implementation summary
- Added `am` CLI support for:
  - `--keep-cr`
  - `--no-keep-cr`
- Added `keep_cr` to persisted am option state (`.git/rebase-apply/options`) and resume override merging.
- Added boolean config resolution helper using `grit_lib::config::parse_bool`, so `am.keepcr` honors Git boolean forms (`true/false`, `yes/no`, `on/off`, `1/0`, etc.).
- Threaded `keep_cr` through patch parsing:
  - mbox parsing now preserves CR bytes in line records when requested.
  - non-keep-cr paths strip CR consistently.
  - line scanning now preserves CRLF boundaries while still recognizing mbox separators and headers.
- Made patch parsing/hunk parsing CR-aware:
  - header detection and hunk parsing now normalize `\r` only for prefix/header matching.
  - payload content itself remains CR-preserving in keep-cr mode.
- Hardened apply matching for `am`:
  - direct apply path normalizes file/hunk comparison according to keep-cr mode.
  - 3-way fallback now also normalizes according to keep-cr mode and validates `index <old>..<new>` preimage OIDs before applying.
