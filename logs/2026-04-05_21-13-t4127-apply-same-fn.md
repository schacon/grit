### Investigation notes
- Reverse-apply failure reproduces with a two-hunk same-file patch (`patch0`) where applying forward succeeds but `git apply -R patch0` fails on second hunk with context mismatch (`expected "e", got "y"`).
- Native Git accepts this sequence, indicating our hunk placement logic diverges in reverse mode for same-file repeated hunks.
- Secondary failure (`A->B, C->A, A->A`) shows current rename/copy/path conflict handling in `grit apply` still rejects a valid sequence that Git accepts.
- Started exploratory adjustments around hunk placement search, but reverted them after no net improvement; no code changes kept for this task yet.

### Fix implemented
- Reverse-apply ordering:
  - Updated `reverse_patches()` to reverse patch list order after flipping hunk directions.
  - This makes `git apply -R` undo later patch sections first, matching native Git semantics for multi-part patches touching the same filename.
- Worktree preflight sequencing guard:
  - Added `precheck_worktree_patch_sequence()` before writing files.
  - The precheck tracks source/target path existence through the patch sequence and fails early for invalid path dependencies with the same "failed to read ... No such file or directory" style expected by tests.
  - Valid rename/copy sequences that rely on initial source snapshots remain allowed.
