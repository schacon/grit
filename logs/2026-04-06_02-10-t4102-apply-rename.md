## Task: t4102-apply-rename

### Claim
- Claimed after completing `t4206-log-follow-harder-copies`.
- Marked in progress while implementing `git apply` rename/copy compatibility.

### Baseline
- `./scripts/run-tests.sh t4102-apply-rename.sh` → `1/5` passing.
- Local failing cases:
  - `git apply --index --stat --summary --apply test-patch` rejected `--apply` as unknown.
  - FILEMODE validation failed after rename apply because `bar` lost executable bit in worktree.
  - reverse/apply-copy cases inherited `--apply` parsing failure.

### Root causes
1. **Missing `--apply` compatibility flag** in `grit apply` CLI parser.
   - Upstream uses `--stat --summary --apply` to show info and still apply.
2. **Worktree mode preservation gap on rename/copy applies**:
   - We wrote patched content to destination but did not preserve executable mode from source-side file/mode metadata, so resulting file mode became `0644`.

### Implementation
- `grit/src/commands/apply.rs`:
  - Added `Args::apply` (`--apply`) as compatibility no-op flag.
  - Changed info-only behavior:
    - `--stat/--numstat/--summary` remain info-only **unless** `--apply` is present.
  - Added source-mode derivation for writeback path:
    - derive mode from `new_mode` when present;
    - else preserve existing destination mode;
    - else fall back to source file metadata;
    - else source-side patch mode (`old_mode`);
    - finally default `0644`.
  - Applied derived mode after writing modified content so rename/copy destination keeps executable bit.
